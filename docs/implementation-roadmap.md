# Roadmap реализации проекта RORTYPE

Обновлено: 2026-06-08

## Цель

Преобразовать текущий Unity-грейбокс в играбельный `vertical slice`, а затем в полноценный прототип игры с:

- `top-down` управлением
- боевым loop
- одной рабочей локацией с задачей, врагами и наградой
- хабом
- переходами между локациями
- прогрессией, экономикой и автосохранением

## Точка старта

Сейчас проект находится в таком состоянии:

- есть один грейбокс уровня `Assets/Game/Scene/Level_1.unity`
- собственной gameplay-логики практически нет
- есть внешний пакет движения/камеры, но он не входит в новый план реализации
- целевой дизайн игры описан значительно подробнее, чем текущая реализация

Из этого следует главный принцип roadmap:

- сначала строить один законченный `vertical slice`
- только потом масштабировать решение на все локации и системы

## Стратегия реализации

Реализацию вести в таком порядке:

1. Сначала фундамент и движение
2. Потом боевой loop в одной тестовой локации
3. Потом награды, враги, интерактивы и UI
4. Потом хаб, переходы и сохранения
5. Только после этого масштабирование на 3 локации и 2 хаба

## Этап 0. Технический фундамент

Цель:

- подготовить проект к системной разработке, а не к разрозненным правкам в сцене

Что сделать:

- собрать проектную структуру папок:
  - `Assets/Game/Scripts/Core/`
  - `Assets/Game/Scripts/Player/`
  - `Assets/Game/Scripts/Combat/`
  - `Assets/Game/Scripts/AI/`
  - `Assets/Game/Scripts/Interaction/`
  - `Assets/Game/Scripts/UI/`
  - `Assets/Game/Scripts/Progression/`
  - `Assets/Game/Prefabs/`
  - `Assets/Game/Scenes/`
- определить:
  - какие сцены являются тестовыми
  - какая сцена является основной gameplay-сценой
- добавить сцены в `EditorBuildSettings`
- зафиксировать базовые `layers`, `tags` и naming rules
- выделить место для ScriptableObject-конфигов:
  - параметры игрока
  - параметры врагов
  - награды
  - encounter-конфиги

Критерий выхода:

- у проекта есть понятная структура, в которой можно безопасно развивать системы

## Этап 1. Движение игрока и камера

Цель:

- получить собственный рабочий `top-down` locomotion-контроллер

Что сделать:

- выполнить `docs/player-movement-roadmap.md`
- собрать:
  - test scene для движения
  - prefab игрока
  - `top-down` камеру
  - sprint
  - dash
  - slope handling
  - fall reset

Критерий выхода:

- игрок стабильно управляется сверху и готов к интеграции с боем

## Этап 2. Combat sandbox

Цель:

- проверить, что персонаж и бой работают как единая система

Что сделать:

- реализовать базовые сущности:
  - `Health`
  - `DamageReceiver`
  - `DamageDealer`
  - `Team/Faction`
- сделать минимальный набор боевых действий игрока:
  - одна базовая атака
  - один активный боевой action или ability
- добавить:
  - hit detection
  - damage application
  - death handling
  - simple hurt feedback
- собрать sandbox-сцену:
  - игрок
  - target dummy
  - один тестовый враг

Критерий выхода:

- игрок может двигаться, атаковать, получать урон и убивать базовую цель

## Этап 3. Core encounter loop

Цель:

- реализовать базовый loop комнаты/локации, вокруг которого строится вся игра

Что сделать:

- создать терминал-портал как старт encounter
- реализовать последовательность:
  - игрок входит в зону
  - активирует терминал
  - запускается задача/событие
  - появляются волны врагов
  - после выполнения условия открывается награда и следующий переход
- определить MVP-условие завершения:
  - убийство всех волн
  - либо убийство элитной цели
- добавить состояние encounter:
  - idle
  - active
  - completed
  - failed/reset

Критерий выхода:

- в одной сцене можно пройти один законченный цикл “активация -> бой -> награда -> переход”

## Этап 4. Враги MVP

Цель:

- получить минимальный, но рабочий набор противников

Что сделать:

- взять 2-3 архетипа из дизайн-памяти:
  - ближний пехотинец
  - стрелок
  - турель
- реализовать для них:
  - обнаружение игрока
  - преследование или удержание дистанции
  - атаку
  - получение урона
  - смерть
- сделать спавнер волн
- ввести базовые настройки сложности:
  - количество врагов
  - частота спавна
  - здоровье
  - урон

Критерий выхода:

- encounter работает не на манекенах, а на реальных врагах

## Этап 5. Награды, ресурсы и лут

Цель:

- замкнуть цикл “бой -> награда -> усиление”

Что сделать:

- реализовать базовые ресурсы:
  - руда
  - опыт
- реализовать дроп с врагов
- сделать MVP интерактивов:
  - одна капсула
  - один сундук
- определить:
  - что игрок получает после encounter
  - как награда читается в UI
- при необходимости сделать очень легкий `inventory-lite`, без полной RPG-системы

Критерий выхода:

- после боя игрок получает осязаемую награду и видит прогресс

## Этап 6. UI и навигация MVP

Цель:

- сделать игру читаемой, а не только технически рабочей

Что сделать:

- собрать HUD:
  - здоровье
  - ресурс
  - опыт
  - cooldown/charge для dash или ability
- добавить минимальные навигационные элементы:
  - маркер терминала-портала
  - маркер награды
  - маркер цели encounter
- сделать placeholder-версию:
  - мини-карты
  - либо компаса
- добавить отдельный подэтап миникарты для игровых сцен:
  - миникарта должна быть scene-bound prefab в правом верхнем углу
  - каждая сцена использует свою статичную картинку карты
  - размер карты в мире должен настраиваться через поля в метрах
  - иконки объектов должны задаваться в prefab самих world-объектов
  - не создавать UI-элементы миникарты через runtime instantiate; использовать заранее подготовленный пул слотов

Критерий выхода:

- игрок понимает, что происходит, куда идти и что он получает

## Этап 7. Хаб MVP

Цель:

- реализовать переход от “одной комнаты” к структуре run-based игры

Что сделать:

- создать простой прототип хаба
- добавить в хаб:
  - торговца
  - кузнеца или placeholder-станцию апгрейда
  - портал/терминал перехода
- реализовать базовый маршрут:
  - завершение локации
  - переход в хаб
  - покупка/улучшение
  - переход в следующую локацию

Критерий выхода:

- игра перестает быть одной encounter-сценой и начинает работать как run-loop

## Этап 8. Прогрессия и автосохранение

Цель:

- перенести дизайн памяти в реальные межлокационные системы

Что сделать:

- реализовать:
  - сохранение прогресса персонажа
  - перенос ресурсов и параметров между локациями
  - автосохранение
- ввести базовое масштабирование сложности:
  - усиление врагов между локациями
- определить, что именно сохраняется:
  - ресурсы
  - уровень/опыт
  - выбранные апгрейды
  - состояние run

Критерий выхода:

- run сохраняет состояние и действительно продолжается между сценами

## Этап 9. Vertical slice первой локации

Цель:

- довести одну локацию до состояния, которое показывает реальную игру

Рекомендуемая первая локация:

- `Лесной водоворот`

Почему:

- у нее уже есть сильная пространственная идея
- она хорошо подходит для проверки top-down навигации, маршрутов и падений

Что сделать:

- переработать текущий грейбокс под правила локации
- добавить:
  - основной маршрут
  - скрытые ответвления
  - одну-две точки награды
  - encounter-зоны
  - безопасные возвраты после падений
- привязать к локации:
  - врагов
  - награды
  - терминал старта
  - финальный переход

Критерий выхода:

- есть одна цельная играбельная локация, которая демонстрирует core gameplay проекта

## Этап 10. Масштабирование на остальные локации

Цель:

- перенести validated loop на весь набор контента

Что сделать:

- после стабилизации первой локации сделать:
  - `Захваченный порт`
  - `Лесной простор`
- не копировать слепо layout первой локации
- сохранять их различия:
  - порт — островная плоскостная навигация
  - простор — большая двухуровневая зона
- адаптировать:
  - encounter pacing
  - типы наград
  - маршрутные ловушки
  - точки возврата

Критерий выхода:

- у проекта есть 3 различимые рабочие боевые локации

## Этап 11. Расширение систем

Цель:

- довести прототип до состояния содержательной playtest-сборки

Что сделать:

- расширить:
  - торговца
  - кузнеца
  - крафт
  - модификации
  - капсулы модификаций
- углубить:
  - enemy variety
  - reward tables
  - progression choices
- при необходимости решить:
  - нужен ли управляемый прыжок в top-down
  - нужен ли aim-facing по курсору

Критерий выхода:

- у игры появляется вариативность, а не только один рабочий loop

## Этап 12. Полировка и QA

Цель:

- подготовить проект к системным плейтестам

Что сделать:

- проверить:
  - стабильность переходов между сценами
  - работу автосохранения
  - корректность respawn
  - отсутствие soft-lock в encounter
  - баланс наград и сложности
- добавить:
  - базовые VFX/SFX hooks
  - читаемые состояния UI
  - debug tools для спавна, encounter и прогрессии

Критерий выхода:

- проект можно отдать на длительный внутренний playtest

## Чего не делать раньше времени

- Не строить все 3 локации одновременно
- Не делать полную экономику до рабочего encounter loop
- Не делать сложную мета-прогрессию до vertical slice
- Не пытаться сразу реализовать весь список врагов из дизайн-документа
- Не смешивать level production и системную разработку в одной хаотичной ветке работ

## Первый практический таск

Если переходить от roadmap к реальной разработке, первый последовательный набор задач должен быть таким:

1. Подготовить проектную структуру `Assets/Game/Scripts/...`
2. Добавить отдельную test scene для top-down движения
3. Реализовать собственный prefab игрока
4. Довести locomotion до состояния, пригодного для combat sandbox

После этого можно переходить к боевому прототипу, не распыляясь на хабы, экономику и все три локации сразу.

## Статус roadmap на 2026-06-08

Фактическое состояние репозитория уже ушло дальше текста ранней версии roadmap:

- Этап 1 по сути выполнен: в проекте есть собственный `top-down` игрок, камера, прыжок, dash, respawn и тестовая сцена
- Внутри combat sandbox уже реализован первый рабочий slice:
  - игрок наносит урон выстрелом и melee
  - есть общий combat-слой с hit info и damage lookup
  - собраны три enemy-prefab для ручной расстановки

### Что считать текущим завершённым подэтапом

- `Combat sandbox` доведён до состояния «игрок может убивать реальных врагов в test/sandbox flow»
- `Враги MVP` частично выполнены:
  - `EnemyShooter`
  - `EnemyMelee`
  - `EnemyExploder`

### Новый подэтап UI-навигации: миникарта

- Следующим принятым подэтапом roadmap считать `мини-карту` внутри `Этапа 6. UI и навигация MVP`.
- Для `Level_1` базовой картинкой карты принят `Assets/Game/Minimap_var_2.png`.
- Базовый размер мира для первой настройки миникарты принят как `500 x 500` метров, но финальные значения должны оставаться редактируемыми в инспекторе.
- Иконки миникарты должны жить в настройках prefab-ов объектов (`TopDownPlayer`, `HAMMER`, `Store`, `Capsule`, `Portal`, `Chest`, `EnemyShooter`, `EnemyMelee`, `EnemyExploder`), а не быть захардкожены в scene logic.
- Подготовка trackable-настроек для этих prefab-ов должна оставаться inspector-driven, с доступными полями `iconSprite` и `iconColor`; при этом для пересборки scene-bound minimap prefab допустим отдельный editor-builder.
- Для игрока на миникарте принят зелёный arrow-marker; для врагов — красный цвет; для остальных интерактивов — синий.

### Уточнение текущего enemy MVP

Для текущего этапа принять такой состав и ограничения:

- стрелок — жёлтая капсула, стрельба раз в 2 секунды в рабочем радиусе
- melee — оранжевая капсула, удар 2 раза в секунду при сближении
- взрывной — фиолетовая капсула, 3 красных предупреждающих мигания перед подрывом
- игрок урон пока не получает
- префабы противников должны существовать как отдельные ассеты для ручной расстановки, а не как сцено-зависимые runtime-объекты

### Следующий практический шаг

Следующим этапом roadmap считать не «создать первых врагов с нуля», а:

1. Подключить enemy-prefab к `PlayerMovementTest` или отдельной sandbox-сцене для быстрой расстановки и прогона.
2. Добавить простой encounter/spawn flow поверх уже готовых enemy prefab.
3. Подготовить scene-bound minimap prefab без runtime-создания новых UI-элементов и с полями размеров карты в метрах.
4. Только после этого переходить к наградам, ресурсам и loop комнаты.
