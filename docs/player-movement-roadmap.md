# Roadmap по реализации движения игрока

Обновлено: 2026-06-04

## Цель

Собрать `v1`-контроллер движения игрока для `RORTYPE`, который соответствует новому принятому направлению:

- 3D `top-down` управление
- собственный проектный контроллер
- без зависимости от импортированного пакета движения
- движение, читаемое сверху и пригодное для боевой игры с исследованием арены

## Жесткие ограничения

- Не использовать `Assets/Plagin/Player Movement and Camera` в реализации движения
- Не использовать этот пакет даже как референс
- Не строить решение поверх его prefab, camera rig, input или motor logic
- Новую систему делать с нуля в проектной зоне

## Текущее состояние

- В репозитории уже есть импортированный пакет `Player Movement and Camera`
- Этот пакет остается в проекте как существующий внешний ассет, но не входит в новый план движения
- Текущая игровая сцена `Assets/Game/Scene/Level_1.unity` пока не содержит собственного подключенного контроллера игрока

## Рабочее решение

Для `v1` принять такой подход:

- Формат управления: `top-down`
- Основа персонажа: `Rigidbody + CapsuleCollider`
- Камера: фиксированная или квазиизометрическая `top-down` камера
- Движение: по плоскости, с проекцией на поверхность
- Направление персонажа:
  - в `v1` персонаж смотрит по направлению движения
  - позже можно развязать facing и movement под боевую систему
- Реализация:
  - только собственные скрипты в `Assets/Game/Scripts/Player/`

## Что входит в v1

- ходьба
- бег
- рывок / ускорение
- поворот персонажа по движению
- устойчивое движение по склонам
- корректная работа на перепадах высоты
- базовый `fall reset`

## Что не входит в v1

- использование старого контроллера пакета
- `third-person` камера
- прыжок как обязательная часть первого инкремента
- `crouch`
- `climb`
- сложная боевая ориентация по курсору
- продвинутые анимационные стейты

## Почему прыжок не в первом инкременте

Для `top-down` управления сначала нужно стабилизировать:

- читаемость перемещения
- качество рывка
- поведение на узких маршрутах
- возврат после падений

После этого отдельно принимается решение, нужен ли проекту именно управляемый прыжок в `top-down`, или задача вертикальности будет решаться:

- рывком
- launch pads
- перепадами высоты без ручного прыжка
- специальными интерактивными маршрутными узлами

## Целевая структура кода

Рекомендуемая папка:

- `Assets/Game/Scripts/Player/`

Рекомендуемые компоненты:

- `TopDownInputAdapter`
  - единая точка чтения ввода
  - не смешивать ввод с мотором
- `TopDownPlayerMotor`
  - движение по плоскости
  - ускорение и торможение
  - sprint state
  - движение по склонам
- `TopDownGroundProbe`
  - проверка поверхности под игроком
  - нормаль поверхности
  - допустимый угол склона
- `TopDownDashAbility`
  - рывок
  - кулдаун
  - окно неуязвимости, если оно понадобится позже
- `TopDownFacingController`
  - разворот по вектору движения
  - подготовка к будущему aim-facing
- `TopDownCameraRig`
  - высота камеры
  - угол наклона
  - follow smoothing
- `PlayerRespawnController`
  - kill volume
  - safe return point
- `PlayerMovementDebug`
  - скорость
  - grounded
  - slope angle
  - dash state

## Архитектурные принципы

- Не смешивать ввод, мотор, камеру и способности в одном скрипте
- Не завязываться на имена объектов или триггеров
- Все параметры движения выносить в инспектор или config-данные
- Сразу отделить `movement state` от будущего `combat state`
- Делать так, чтобы позже можно было добавить:
  - aim по курсору
  - круговой стрейф
  - combat-lock facing

## Этапы реализации

## Этап 0. Зафиксировать top-down спецификацию

Оценка: `0.5 дня`

Что сделать:

- Зафиксировать, что основной формат игрока в проекте: `top-down`
- Выбрать тип камеры для `v1`:
  - фиксированная top-down
  - фиксированная с наклоном
  - мягко следящая без ручного вращения
- Зафиксировать базовый набор параметров:
  - walk speed
  - sprint speed
  - acceleration
  - deceleration
  - rotation speed
  - dash distance
  - dash cooldown
  - max slope angle

Критерий выхода:

- есть понятная спецификация `v1`, без конфликтов с `TPS`

## Этап 1. Тестовая сцена движения

Оценка: `0.5-1 день`

Что сделать:

- Создать отдельную сцену для теста движения
- Собрать на ней:
  - плоский участок
  - широкую арену
  - узкий коридор
  - повороты на 90 градусов
  - подъем и спуск
  - островки и зоны падения
- Добавить safe respawn point

Критерий выхода:

- есть отдельная сцена, где можно быстро тестировать top-down locomotion без шума остального проекта

## Этап 2. Базовый top-down мотор

Оценка: `1-2 дня`

Что сделать:

- Создать новый prefab игрока
- Настроить:
  - `Rigidbody`
  - `CapsuleCollider`
  - root object
  - visual child
- Реализовать:
  - чтение осей движения
  - движение по плоскости
  - ускорение и торможение
  - sprint
  - разворот персонажа по вектору движения
  - ограничение паразитной инерции

Критерий выхода:

- игрок стабильно двигается по арене
- управление отзывчивое и читаемое сверху

## Этап 3. Grounding и склоны

Оценка: `1 день`

Что сделать:

- Реализовать `ground probe`
- Добавить распознавание:
  - поверхности
  - края
  - недопустимого уклона
- Настроить проекцию движения на наклонную поверхность
- Исключить:
  - дрожание на стыках
  - подскоки на ребрах
  - неконтролируемое скольжение

Критерий выхода:

- игрок предсказуемо движется по рельефу и не ломается на простом вертикальном грейбоксе

## Этап 4. Рывок / ускорение

Оценка: `1 день`

Что сделать:

- Реализовать отдельную способность `dash`
- Зафиксировать рывок в направлении текущего движения
- Если игрок стоит:
  - использовать последнее направление
  - либо запретить рывок без входного вектора
- Настроить:
  - дистанцию
  - длительность
  - кулдаун
  - поведение возле стен

Критерий выхода:

- рывок улучшает навигацию и бой, но не позволяет ломать маршруты или пролетать сквозь коллизии

## Этап 5. Падения и возврат

Оценка: `0.5-1 день`

Что сделать:

- Добавить `kill volume`
- Добавить систему safe respawn
- Определить:
  - как выбирается последняя безопасная точка
  - что происходит при падении с маршрута
  - сбрасывается ли скорость и состояние рывка

Критерий выхода:

- игрок не уходит в soft-lock после падения с маршрута

## Этап 6. Камера top-down

Оценка: `1 день`

Что сделать:

- Настроить камеру под читаемость боя и маршрутов
- Зафиксировать:
  - высоту
  - tilt
  - дистанцию
  - smoothing
- Проверить, что камера:
  - не теряет игрока
  - не режет читаемость краев
  - не конфликтует с узкими участками

Критерий выхода:

- камера и движение ощущаются как одна система, а не как два отдельных костыля

## Этап 7. Подготовка к боевой интеграции

Оценка: `0.5-1 день`

Что сделать:

- Ввести параметры для Animator:
  - move speed
  - is moving
  - dash
- Подготовить точку расширения под будущий facing mode:
  - face by move
  - face by aim
- Не реализовывать aim-систему целиком, только заложить интерфейс

Критерий выхода:

- мотор не придется переписывать, когда начнется интеграция с боем

## Что не стоит делать сейчас

- Не возвращаться к `TPS` как промежуточному этапу
- Не использовать старый пакет даже для копирования параметров
- Не делать одновременно `top-down` и `twin-stick` боевую ориентацию
- Не добавлять прыжок до стабилизации base locomotion
- Не смешивать camera rig с motor logic

## Минимальный порядок реализации

1. Зафиксировать top-down спецификацию
2. Собрать тестовую сцену движения
3. Создать новый prefab игрока
4. Реализовать базовый мотор
5. Сделать grounding и slope handling
6. Добавить dash
7. Добавить fall reset
8. Настроить top-down камеру
9. Подготовить интеграцию с анимацией и боем

## Практический следующий шаг

Если переходить от планирования к реализации, первый рабочий таск должен быть таким:

`Собрать отдельную test scene и новый top-down prefab игрока с базовым planar locomotion, sprint и поворотом по движению, без dash и без прыжка.`
