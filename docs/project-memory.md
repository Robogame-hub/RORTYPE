## Целевое видение по дизайн-документу

- Игра строится вокруг прохождения основных локаций с переносом прогресса персонажа между ними
- Сохранения: автосохранение
- Прогрессия:
  - прогресс персонажа сохраняется между локациями
  - монстры усиливаются при переходе между локациями
  - локации сменяются последовательно
- Структура мира:
  - 3 основные боевые локации
  - 2 под-локации/хаба
  - основная организация: смесь хаб-системы и линейного прохождения
- Возврат:
  - вернуться можно только через хаб между локациями
- Физика мира:
  - разрушения нет
  - физика почти отсутствует, кроме гравитации в прыжке героя

## Целевые локации

### 1. Лесной водоворот

- Закрытая боевая локация с рандомными основными задачами
- Форма: спиралевидная гора/холм
- Ключевая идея:
  - игрок поднимается от берега по широкой спиральной тропе наверх
  - у уровня должны быть скрытые и альтернативные пути по краям
  - падение с основного пути должно вести к более длинному/тяжелому маршруту с возможностью вернуться и получить дополнительный лут
- Визуальная драматургия:
  - у подножия много растительности, дерева, бамбука и природных троп
  - выше растительность тускнеет, становится болотной и темной
  - ближе к пику появляются панели, трубы и другие техно-элементы
  - вершина оформлена как внутренний котлован с металлической отделкой и оголенными горными массами
- Особенность:
  - выраженная вертикальная мобильность
  - нужны перепрыгивания, пропасти, прыжковые участки и нижние обходные тропы

### 2. Захваченный порт

- Закрытая боевая локация с рандомными основными задачами
- Форма: близкая к прямоугольной
- Ключевая идея:
  - уровень состоит из плоских островов
  - перемещение между островами строится на прыжках и джамп-пазлах
  - при падении в воду игрок возвращается на ближайшую безопасную точку
- Визуальная драматургия:
  - разрушенные острова
  - мало растительности
  - серо-болотная и коричнево-темная палитра
  - техно-постройки должны объяснять разрушение прежнего массива острова
- Особенность:
  - вертикальная мобильность почти отсутствует
  - скрытые мелкие островки вокруг основного маршрута используются как места для лута

### 3. Лесной простор

- Закрытая боевая локация с рандомными основными задачами
- Форма: смесь прямоугольника и круга
- Ключевая идея:
  - уровень ощущается как двухэтажный
  - первый этаж: длинная плоская природная зона
  - второй этаж: верхняя секция с техно-застройкой
  - переходы на второй этаж идут по боковым мостам, один мост цельный, второй разрушен и превращен в большой джамп-пазл
- Особенность:
  - уровень больше первых двух
  - альтернативные пути не требуются так сильно, как на первом уровне

## Принятое рабочее решение: движение игрока

- Для реализации движения игрока создан отдельный roadmap: `docs/player-movement-roadmap.md`
- Рабочая база для `v1`:
  - `top-down` контроллер
  - `Rigidbody + CapsuleCollider`
  - модульная проектная реализация в `Assets/Game/Scripts/Player/`
- Импортированный пакет `Player Movement and Camera` не использовать в реализации движения и не использовать даже как референс
- Для первой версии движения не считать обязательными:
  - прыжок
  - `crouch`
  - `climb`
- Текущее реализованное состояние:
  - `TopDownInputAdapter`
  - `TopDownGroundProbe`
  - `TopDownPlayerMotor`
  - `TopDownFacingController`
  - `TopDownCameraRig`
  - `PlayerRespawnController`
  - тестовая сцена `PlayerMovementTest` с игроком и камерой
  - `EditorBuildSettings.asset` указывает на `PlayerMovementTest` как на текущую сцену входа для теста движения
  - для `TopDownGroundProbe` принято рабочее ограничение под Unity `6000.3.10f1`: не сериализовать `LayerMask` как `-1` в сцене/prefab; значение `0` трактуется кодом как fallback на слой `Ground`, а если такого слоя нет - на `Default`, чтобы не ловить prefab import error `Failed to convert -1 to a unsigned 32 bit int` и не считать любые коллайдеры землей
  - для текущего `v1` принята коррекция физики движения: `TopDownGroundProbe` должен игнорировать собственный collider игрока, `TopDownFacingController` не должен крутить `transform` динамического `Rigidbody` напрямую в `LateUpdate`; `groundedStickForce` больше не используется, когда игрок соприкасается с `Ground`, вертикальная скорость сбрасывается в `0`, а дополнительная гравитация применяется только в воздухе
  - для текущего тестового контроллера `TopDownPlayerMotor` упрощен расчет planar velocity: `v1` копит собственную `planarVelocity`, перемещает физический root игрока по земле напрямую через `Rigidbody.MovePosition` и snap по `Y` к `GroundPoint`, а видимая капсула вынесена в дочерний `Visual` и сглаживается в `LateUpdate`; базовые test-значения повышены до `walkSpeed = 10`, `sprintSpeed = 15`, `acceleration = 60`, `deceleration = 80`
  - в текущем рабочем состоянии `TopDownPlayerMotor` уже содержит прыжок и рывок внутри одного мотора, а не через отдельные ability-компоненты: прыжок работает на собственной `verticalVelocity`, `dash` держит фиксированную planar-скорость на время `dashDuration`, `Rigidbody.useGravity` должен быть отключен, потому что `Y` контролируется кодом
  - текущий боевой прототип добавлен поверх того же контроллера: `TopDownFacingController` разворачивает персонажа к курсору мыши на горизонтальной плоскости, стреляет красными runtime-сферами через `TopDownProjectileSphere` по `Left Mouse Button`, а по `Right Mouse Button` запускает ближний бой двумя красными runtime-сферами-кулаками
  - ближний бой в текущем прототипе реализован как чисто controller-визуал без урона: при нажатии `Right Mouse Button` выполняется одиночный удар, при удержании удары продолжаются автоматически и кулаки чередуются левый/правый; активный кулак выдвигается по направлению взгляда к мыши и немного скейлится во время удара
  - у `Visual` игрока поверх стрельбы и ближнего боя оставлен общий `bounce`-эффект через squash/stretch масштаба; реализация не использует runtime-child `FeedbackVisual`, а масштабирует сам `visualRoot` либо явно назначенный `feedbackTransform`, потому что `TopDownPlayerMotor` сглаживает у `visualRoot` только позицию и не затирает scale
  - текущие тестовые кнопки контроллера: `WASD` движение, `LeftShift` sprint, `Space` прыжок, `LeftControl` dash, `R` respawn, `Left Mouse Button` стрельба, `Right Mouse Button` ближний бой
- Что еще не реализовано в этом инкременте:
  - урон/попадания по врагам и полноценная боевая интеграция
  - анимационная интеграция

## Принятое рабочее решение: combat sandbox и враги MVP

- В проекте добавлен отдельный слой боевой логики и AI, не основанный на скриптах игрока:
  - `Assets/Game/Scripts/Combat/CombatHitInfo.cs`
  - `Assets/Game/Scripts/Combat/CombatUtility.cs`
  - `Assets/Game/Scripts/Combat/TransientScaleEffect.cs`
  - `Assets/Game/Scripts/AI/EnemyCapsuleController.cs`
  - `Assets/Game/Scripts/AI/EnemyProjectileSphere.cs`
- Текущий прототип игрока больше не ограничен только визуалом атак:
  - `TopDownProjectileSphere` теперь может наносить урон `IDamageable`-целям
  - `TopDownFacingController` наносит melee-урон через overlap-hit, а не только через визуальные кулаки
- Принятое ограничение текущего этапа:
  - игрок по-прежнему не получает урон от противников
  - атаки врагов сейчас нужны для читаемого поведения и фидбэка, а не для полноценного двустороннего damage-loop
  - при этом атаки врагов теперь могут заметно, но не слишком сильно отталкивать игрока через отдельный канал knockback без снятия HP
- В проекте созданы реальные prefab-ассеты трёх капсульных врагов для ручной расстановки:
  - `Assets/Game/Prefabs/Enemies/EnemyShooter.prefab` — жёлтый стрелок, частая прицельная стрельба и круговой залп раз в 3 секунды, `HP = 5`
  - `Assets/Game/Prefabs/Enemies/EnemyMelee.prefab` — оранжевый melee-враг, удар 2 раза в секунду вблизи, `HP = 10`
  - `Assets/Game/Prefabs/Enemies/EnemyExploder.prefab` — фиолетовый взрывной враг, перед подрывом 3 раза мигает красным, `HP = 3`
- Общий фидбэк текущего enemy MVP:
  - при атаке враги делают `bounce` через squash/stretch
  - при попадании игрока враги 3 раза мигают белым
  - над врагами теперь показывается runtime-HP bar с текущим/максимальным HP; полоска увеличена вдвое и использует красный fill
  - при попадании по врагам появляются вылетающие цифры полученного урона
  - melee-враг теперь использует визуальные runtime-кулаки по образцу игрока, но на собственной AI-логике
- Для prefab-пайплайна принят editor-builder:
  - `Assets/Game/Editor/EnemyPrefabBuilder.cs`
  - prefab- и material-ассеты собираются через Unity batchmode или меню `RORTYPE/Build Enemy Prefabs`

## Обновление реализации на 2026-06-08: NavMesh и spawn zone

- Текущая реализация врагов переведена на `NavMeshAgent`:
  - `EnemyCapsuleController` больше не ходит прямым `Rigidbody.MovePosition`
  - враги патрулируют между двумя navmesh-точками вокруг своего якоря, пока не обнаружат игрока
  - радиус первичного обнаружения игрока принят `25` метров
  - после захвата цели враг удерживает бой до удаления игрока дальше `30` метров
- Для стрельбы принят единый рабочий лимит дальности `20` метров:
  - у игрока effective lifetime снаряда теперь дополнительно ограничивается расчётом `distance / speed`
  - у стрелка-врага используется такой же подход через effective lifetime
  - `EnemyShooter` переведён на `shooterAttackRadius = 20`
- В проект добавлена заготовка зоны спавна врагов:
  - `Assets/Game/Scripts/AI/EnemySpawnZone.cs`
  - `Assets/Game/Scripts/AI/EnemySpawnPoint.cs`
  - зона активируется, когда игрок входит в trigger-объём
  - по умолчанию лимит активных врагов в зоне `25`
  - спавн идёт со случайным интервалом в диапазоне `8-12` секунд вокруг целевых `10` секунд
  - тип врага выбирается по весам: melee `50`, shooter `30`, exploder `20`
- Для prefab workflow добавлен builder зоны спавна:
  - `Assets/Game/Editor/SpawnZonePrefabBuilder.cs`
  - целевой prefab-путь: `Assets/Game/Prefabs/Spawning/EnemySpawnZone.prefab`
- Runtime HP bar врагов скорректирован под новое требование:
  - высота увеличена в 2 раза
  - длина уменьшена в 2 раза

## Принятое рабочее решение: combat sandbox revision 2 на 2026-06-08
